# MIGRATION.md — План перехода к рабочей среде из окон и пресетов

> **Что это.** Пошаговый план превращения текущего `index.html` (единственный layout V2 =
> неявный пресет «Прогностика») в архитектуру из `WORKSPACE_CONCEPT.md`.
> `WORKSPACE_CONCEPT.md` отвечает на «что и почему», этот документ — на «в каком порядке и как»,
> с привязкой к реальным функциям. Строки указаны на момент написания (файл ~8959 строк) — сверять.

---

## Принципы миграции

1. **Никакого big-bang.** Каждый инкремент — отдельный, самодостаточный, в идеале отдельный коммит.
2. **Поведение сохраняется на каждом шаге** (кроме шагов, где новая фича добавляется явно).
   Критерий успеха инкрементов-рефакторов: «всё работает ровно как раньше».
3. **Глобалы остаются рантаймом, контекст — сериализованной формой** (на первых шагах).
   Не переписываем `renderChart`/`switchToMode` — они продолжают читать глобалы.
   Меняем только то, как состояние *упаковывается* в кейс/вкладку.
4. **Проверка в браузере после каждого шага**: открыть карту → переключить режимы (Натал/Транзит/
   Прогр./Дирекц.) → переключить вкладки → перезагрузить страницу (restoreTabs). Поведение идентично.
5. **Толерантность к отсутствию данных** (как уже сделано с `charts.pinned`/`last_opened_at`):
   новые поля Supabase опциональны до миграции БД.

## Инкремент 1 — Контекст-объект (рефактор `st`) ✅ СДЕЛАНО

> Реализован: `captureState`/`applyState`/`stateToView` в `index.html` теперь оперируют
> структурой `{ subjects, layers, T, ui }`. Формат localStorage (`view`) не изменился — миграция не нужна.
> Проверено в браузере: режимы Натал/Транзит/Прогр./Дирекц., свап вкладок, перезагрузка — без ошибок.

**Цель.** Заменить плоский мешок `st` структурированным контекстом, не меняя рантайм-глобалы.

**Что меняется (точечно):**
- `captureState()` (`:3589`) строит структуру вместо плоского объекта:
  ```js
  context = {
    subjects: [{ id, role: 'natal', data: natalData }],   // самосущие
    layers:   [{ method: currentMode, driven: true }],    // производные (transit/progressed/direction)
    T: currentTransitJD,                                  // ось времени (null = сегодня)
    ui: { tlState: layout2TlState, chartExpanded, tlYearScale, moonBandCollapsed,
          expandedRows, fastCache, hiddenTPlants, cachedTimeline,
          tlPastYears, tlFutureYears }
  }
  ```
- `applyState(s)` (`:3599`) распаковывает контекст обратно в существующие глобалы
  (`natalData = ctx.subjects.find(s=>s.role==='natal').data` и т.д.). **Рендер не трогаем.**
- `snapshotActiveTab()` (`:3615`), объект вкладки (`:3717`), `openNewTabFromForm` (`:3737`) — кладут/читают `t.st = context`.
- Бамп версии формата в `persistTabs`/`restoreTabs` + миграция старого плоского `st` при чтении.

**Подводные камни.** `natalData` мутируется в нескольких местах — следить, чтобы ссылка в `subjects[0].data`
не разъезжалась с глобалом. На первом шаге проще: контекст собирается из глобалов в момент `captureState`,
а не держится «живым».

**Риск.** Низкий–средний. Чистая сериализация, локализована в ~4 функциях.

**Проверка.** Открыть карту, погонять режимы и вкладки, перезагрузить. Старые вкладки в localStorage
должны мигрировать без потери. Поведение идентично.

**Открывает.** Явный список `subjects` (под синастрию), явные `T` и `ui.zones`.

## Инкремент 3a — Палитра новой вкладки ✅ СДЕЛАНО (v1)

> Кнопка `+` в таб-баре открывает **пустую вкладку с палитрой** (WORKSPACE_CONCEPT §8) вместо
> модалки формы. Вкладка получила `kind: 'palette' | undefined` (undefined = кейс); палитра
> персистится в localStorage и восстанавливается. Контент: команды-конструкторы (активна
> «Новая карта»; Синастрия/Хорар/Элекция — заглушки «скоро») + поиск по `_homeCharts`
> (pinned → last_opened_at, стрелки/Enter) → выбор открывает кейс **в этой же вкладке**
> (`kind` снимается); если кейс уже открыт в другой вкладке — фокус туда, осиротевшая палитра
> закрывается. «Новая карта» открывает модалку формы; `calcBtn` при активной палитре превращает
> её вкладку в кейс (`computeIntoPaletteTab`), а не создаёт новую. Snapshot/hash для палитры
> не пишутся (`snapshotActiveTab` пропускает, hash очищается).
>
> Осталось (при реализации типов кейсов): включить конструкторы синастрии/хорара/элекции,
> Cmd+K — палитра модалкой для перехода без новой вкладки.

## Инкремент 5 — Синастрия (2 субъекта) ⬅ ПЕРВАЯ НАСТОЯЩАЯ НОВАЯ ЗАДАЧА

**Цель.** Композитный кейс на два субъекта: совмещённое биколесо A+B, межкарточные аспекты, окно данных
показывает 2 карточки (бесплатно из инкр. 4), таймлайн выключен.

**Что меняется.**
- Конструктор в палитре «Создать синастрию» (выбор кейса A и B из библиотеки).
- `context.subjects` содержит 2 элемента (роли `A`/`B`, оба `anchored`).
- Новый путь рендера совмещённого колеса — переиспользовать структуру `renderBiwheel` (`:5244`),
  но оба кольца натальные (A внутри, B снаружи), без `driven`-логики.
- Сетка кросс-аспектов — переиспользовать `buildAspectGrid` (A против B).
- Таймлайн отключён (состояние слота в пресете).

**Риск.** Высокий — новый рендер + двухсубъектная модель. Поэтому идёт ПОСЛЕ инкр. 1 (контекст) и 4 (данные).

**Проверка.** Создать синастрию из двух кейсов библиотеки; 2 карточки; кросс-аспекты; контроллер `T` недоступен.

**Открывает.** Доказательство тезиса «пресеты — это данные, а не код».
